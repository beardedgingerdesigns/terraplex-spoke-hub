## Why this is library-worthy
A **product-attached downloads / resource library** product-block — the page lives at `/<product-slug>/downloads` and surfaces every artifact an owner of that product needs after purchase: companion mobile apps, OS-paired desktop software and drivers, and a flat manuals list. Fills a real gap: the existing 17 product-block patterns (including the new [zone-categorized accessory catalog](./refero-0d9a0f91-zone-categorized-accessory-catalog.md) added this run) handle pre-purchase and accessory-merchandising cases — "which model? / which spec? / which bundle? / which accessory zone?" — but none answer **"I already own the [R-32]; where do I get the latest firmware, the user manual, and the iOS app?"** This is the post-purchase ownership pattern. Functional, utilitarian, but specifically valuable for Terraplex dealers running [R-32](../../../products/r-32.md) and [I-19](../../../products/i-19.md): both products ship firmware on an ongoing cadence (the [2026-05-12 FCC waiver digest](../../../research/digests/2026-05-12.md) makes this even more concrete — the firmware-update pipeline is regulated through Jan 2029), both have companion mobile apps for ground-station control and mission planning, and both need accessible PDFs for the [Part 137](../../../content/compliance-program.md) compliance audit trail. The downloads-library composition — `Apps row` → `Softwares & Drivers paired-OS rows` → `Manuals flat list` — is the load-bearing structure that scales from 3 to 30+ artifacts without losing scannability.

## What to take
- **Tabbed sub-nav anchored to the product.** The page opens with a horizontal product-context strip carrying the product nameplate at the left (`DJI Ronin 4D` → for a dealer `R-32` or `I-19`) followed by a small in-product nav (`Walking with Masters`, `Lens Compatibility`, `Specs`, `FAQ`, `Video`, `Buy RAW License Key`, `Downloads`, `Sample Footage`, plus a right-aligned `Buy Now` pill). The active tab (`Downloads`) is bolded and underlined. This signals "you're inside one specific product's resource library, not the dealer's global downloads page" — important because customers often Google `[product name] firmware` and land here directly. For a dealer the equivalent is `R-32` → `Specs` / `Manuals` / `Firmware` / `Training` / `Part 137 Compliance` / `Buy Now`.
- **Page-level "Downloads" heading, no marketing copy.** A single `~28px` left-aligned page heading `Downloads` sits at the top of the page-body — no subhead, no hero band, no marketing sentence. The utilitarian register IS the credibility: customers here are not browsing, they're retrieving a specific file. Marketing copy would feel like friction. Compose this with the [hardware-lineup dark hero](../../hero/research/refero-0945a36c-hardware-lineup-dark-hero.md) or the [pdp-product-detail hero](../../hero/research/refero-d4c69c08-pdp-product-detail-hero.md) on the parent PDP, where marketing copy belongs.
- **`Apps` section: 2-column app-store rail + QR code.** First section (titled `Apps` in `~16–18px` bold sentence-case). Each app row has the app icon at the left (`~48×48px` rounded square), the app name in bold sentence-case (`DJI Store` → for a dealer `R-32 Mission Planner` / `I-19 Spray Controller`), a 1-line description in muted-gray (`The DJI Store app lets you effortlessly shop for DJI products and enjoy exclusive discounts.` → for a dealer `Mission planning, telemetry, and Part 137 logbook in one app.`), followed by paired download badges (`Download on the App Store` + `Get it on Google Play`) sitting side-by-side. Beneath each badge, a `iOS Version v5.3.0 — Requires iOS 12.0 or above` / `Android Version v5.3.0 — Requires Android 7.0 or above` micro-caption gives the version constraint without a separate detail page. To the right of the paired badges, a `~120×120px` QR code lets a desktop customer scan-and-install on phone without re-typing the store URL. The QR code is the pattern's smartest bridge: it ties the desktop downloads page to the mobile install flow without a separate "share to phone" UX. For a dealer with a real mobile app this is a meaningful capability surface; for a dealer without one, omit the section entirely (better than a stub).
- **`Softwares & Drivers` section: per-package paired-OS rows.** Second section. Each software package (firmware tool, LUT pack, calibration utility, asset library) anchors its own sub-block with a `~32×32px` icon at the upper-left, the package name in bold (`DJI Assistant 2 (Ronin Series)` → for a dealer `R-32 Service Tool` / `I-19 Spray Calibration Utility`), and a 1-line description below the name. Beneath that, the row splits into **two columns: macOS left, Windows right**, each with the same shape — OS-glyph at the leftmost position (`` Apple `` / `` Windows `` logo, `~24×24px`), version label (`Mac Version V2.1.6`), release date below the version in muted-gray (`2023-03-07`), and the actual download link as a small filename pill at the bottom (`.pkg ⬇` for macOS, `.exe ⬇` and `.zip ⬇` for Windows — Windows often ships two formats; macOS one). The paired-OS layout is the pattern's most reusable move: it solves the cross-platform tax of "which OS am I on, again?" by presenting both side-by-side with no tab-switching. For a dealer this scales to whatever OSes their software touches — adding a `Linux ⬇` third column for command-line tools is structurally fine.
- **Release notes inline, not behind a click.** Each software package row ends with a flat single-line `[Package] Release Notes V[X.Y.Z]` link with a date on the right and a `pdf ⬇` download glyph at the rightmost edge. Release notes are themselves a PDF download, not a modal or a separate page — owners doing a compliance audit need the release notes as a file to attach to their records, so making them a click-to-open page would force them to print-to-PDF anyway. For a Terraplex dealer this matters because [Part 137](../../../content/compliance-program.md) operators need to demonstrate they're flying current firmware; a downloadable release-note PDF IS the evidence.
- **`Manuals` section: flat list with right-aligned date + format glyph.** Third section. A flat vertical list of every PDF the product owner could ever need — User Manual, Quick Start Guide, Safety Guidelines, accessory-specific guides (`DJI Ronin 4D DJI PROSSD Mount - User Guide v1.0`), and any product-information PDFs for compatible accessories (`DJI Ronin 4D DJI LiDAR Range Finder - Product Information v1.0`). Each row is a single horizontal line: manual title in regular-weight sans-serif at the left (`~14–15px`), release date in muted-gray at the middle-right (`2023-03-07`), `pdf ⬇` glyph at the rightmost edge. Hairline rule beneath each row. No card chrome, no thumbnail previews, no category sub-headers — the list is intentionally flat because manual count grows over the product's life and any sub-categorization eventually becomes stale. A dealer's R-32 list might start at 5 PDFs (User Manual, Quick Start, Safety, Spreader Hopper Guide, Battery Care) and grow to 20+ as accessories ship.
- **Footer is the dealer's standard footer, not a section-specific footer.** Critical structural decision: the downloads page does NOT close with a CTA band, a "Need help?" callout, or a related-products row. The page ends when the manual list ends, then the page hits the standard site footer. Marketing or upsell at the end of a downloads page would feel intrusive to a customer just trying to grab a file; the silence is intentional. If the dealer wants to route stuck customers to support, the dealer's standard help-center link in the global footer is sufficient.
- **The product-context strip persists at the top of the page.** The product nameplate + in-product nav strip stays visible (sticky or near-sticky) so a customer who scrolled deep into the manuals list can jump back to `Specs` or `Buy Now` without bouncing to the site nav.

## What NOT to take
- **Don't bury individual artifacts behind a "click to expand" or modal.** The pattern's affordance is "every download on one scrollable page; Cmd-F can find the SKU you need." Forcing expansion or modals breaks Cmd-F and forces the customer to remember which collapsed section had the file. The DJI page deliberately ships everything inline.
- **Don't sub-categorize the manuals list when count is low.** With fewer than ~12 manuals, a flat list scans faster than a categorized one. Only when count exceeds ~20 should the list split into named sub-groups (`Operator Manuals`, `Accessory Guides`, `Safety & Compliance`); even then, keep each sub-group flat within itself.
- **Don't replace the QR code with a "Send to phone" form-field UX.** Form fields require the customer to type their own phone number / email and accept an SMS or email; that's friction for what should be one scan. The QR is universal, no auth required, no privacy disclosure.
- **Don't add release notes as a click-through page when an audit-able PDF would do.** Release notes live alongside the firmware downloads as another PDF on the same row — owners who need to attach release notes to compliance records get them as a file directly.
- **Don't introduce sales pressure on the downloads page.** No "Upgrade to Pro" banner, no "Have you renewed your service plan?" callout, no "Featured accessory" sidebar. The downloads page is post-purchase ownership space; marketing chrome here reads as "the dealer doesn't respect me as an existing customer." If renewal upsell is needed, surface it on the parent PDP, on the account-dashboard, or in an email — not here.
- **Don't paginate the manuals list.** Pagination breaks Cmd-F and forces the customer back to navigation chrome. If the list is genuinely too long, prefer the sub-categorization escape valve.
- **Don't omit the version constraint micro-caption under app-store badges.** Customers on older iOS / Android need to know IMMEDIATELY whether the latest app version supports their phone; without the `Requires iOS 12.0 or above` line they'll install, fail, and re-google.

## Adaptation notes (cross-archetype)
- **CLEAN**: native fit (the DJI source is essentially CLEAN). White / `~#F7FAFC` page background, charcoal type, brand-navy on link pills and the active in-product nav tab, muted-gray for dates and version constraints. Hairline rules between manual rows in `~#E8E8E8`. The QR code stays charcoal-on-white. App-store badges use the official Apple / Google badge artwork (don't restyle these — Apple's badge guidelines require unmodified usage). See [CLEAN archetype](../../../archetypes/clean.md).
- **FORGE**: invert to a deep-charcoal / near-black page (`~#161616`). White type for headings and manual titles, muted-gray for dates, ember-orange (`#FF6A1F`) on the active in-product nav underline + the `pdf ⬇` / `.zip ⬇` download glyphs (so the file-format pills become the single ember accent across the page). The OS-glyph rendering inverts to white. Hairline rules between manual rows go charcoal-on-near-black. QR code inverts to white-on-charcoal. App-store badges stay in their official treatments (do not restyle). See [FORGE archetype](../../../archetypes/forge.md).
- **TACTICAL**: precision-sans typography (`JetBrains Mono` for filenames and dates; Inter for descriptions), near-black page, steel-cyan (`#4FB2D9`) on active nav + download-glyph accents. Version constraints render with mono digits for legibility. Corner-bracket motif (`⌐` glyph) at the leftmost margin of section headers. Pairs naturally with the [interactive product-demo hero](../../hero/research/refero-d6176d7e-interactive-product-demo-hero.md) on the parent PDP — both share the precision-engineering register. See [TACTICAL archetype](../../../archetypes/tactical.md).
- **TERRAIN**: cream-beige page (`~#F7F1E5`) with deep-charcoal type, terrain-clay (`#C25A2A`) accent on the active in-product nav and the `pdf ⬇` glyphs. Hairline rules in warm-stone gray. The page-heading `Downloads` can gain a small contour-line eyebrow motif above it (single line, low opacity) — the pattern's only decorative concession. QR code stays charcoal. Compose with the [topographic layered-bg hero](../../hero/research/refero-ae84b62a-topographic-layered-bg.md) on the parent PDP. See [TERRAIN archetype](../../../archetypes/terrain.md).
- **FIELD**: warm-cream page (`~#F4EFE7`) with deep-charcoal type, warm-amber on active nav and download glyphs. Section headers in a slightly editorial serif (the only typographic adjustment — body and filenames stay sans). The QR-code section can gain a tiny "Scan with your phone — works in the field, no signal needed once installed" line in muted-warm-gray (FIELD operators care about offline behavior; flagging it here is value-add, not marketing). Pairs with the [hands-on craft hero](../../hero/research/refero-be842c1e-hands-on-craft-hero.md). See [FIELD archetype](../../../archetypes/field.md).
- **PATRIOT**: cream off-white (`~#F8F4ED`) page, deep-navy type, barn-red on the active in-product nav, deep-navy on download glyphs. The page-heading `Downloads` can sit in a heritage serif at slightly larger weight than the other archetypes (PATRIOT's editorial register). Manuals list stays utilitarian — no editorial flourish in the list itself, only in the heading. The release-notes PDFs gain explicit `Release Notes — vX.Y.Z — Issued [date]` framing (PATRIOT's documentary-language register reads as "official record"). See [PATRIOT archetype](../../../archetypes/patriot.md).

## Source brief
[DJI's `/ronin-4d/downloads`](https://www.dji.com/ronin-4d/downloads) — DJI surfaces the complete post-purchase resource library for the Ronin 4D cinema gimbal-camera system in a three-section composition (Apps row with paired App Store + Play Store + QR code; Softwares & Drivers section with per-package macOS / Windows paired-OS rows + version + release date + filename-pill downloads + release-notes PDF; Manuals section as a flat list with right-aligned date + `pdf ⬇` glyph per row). The page sits behind the product's in-product `Downloads` tab so customers Googling `ronin 4d firmware` land here directly. The structural move (product-attached + utilitarian heading + three named sections in fixed order + paired-OS columns + flat manuals list + no marketing chrome) is reusable for any Terraplex dealer maintaining firmware, calibration utilities, mobile apps, and PDF manuals across [R-32](../../../products/r-32.md) and [I-19](../../../products/i-19.md) — particularly important for [Part 137](../../../content/compliance-program.md) operators who need a downloadable evidence trail of every firmware version they've flown.
